# Interview Pocket 面试讲解指南

本文档帮助你在面试中清晰地讲解 Interview Pocket 项目。

## 1. 30 秒项目介绍

> Interview Pocket 是我做的一个本地桌面面试复习工具，主要解决 Markdown 面试资料在题量变多后难搜索、难跟踪复习状态、难沉淀错题的问题。
>
> 它基于 Electron + Vue 3 + TypeScript + sql.js 实现，支持 Markdown 导入、题库搜索、来源筛选、复习状态筛选、普通复习、错题复习、突击模式和复习统计。
>
> 所有数据存储在本地，不依赖云服务，离线可用。

## 2. 2 分钟项目介绍

### 背景

我在准备技术面试时，积累了大量 Markdown 格式的面试资料，涵盖 Java 后端、AI 应用后端、项目经验等方向。

### 痛点

随着题量增长到 150+ 道，传统 Markdown 复习方式暴露出几个问题：

- **检索困难**：题目多了以后，想找特定知识点的题目很麻烦。
- **状态不明**：不知道哪些题复习过，哪些还没看。
- **错题流失**：做错的题没有沉淀，下次还是会错。
- **突击无序**：面试前想快速过一遍重点题，没有独立模式。
- **来源混乱**：多份题库文件，重复导入、旧版本难清理。

### 技术选型

- **Electron**：跨平台桌面框架，个人项目只需在 Windows 运行，但技术栈本身具备跨平台能力。
- **Vue 3 + TypeScript**：前端框架，Composition API 适合组织复杂交互逻辑。
- **sql.js**：纯 JS 实现的 SQLite，无需编译原生模块，数据保存为单个 `.db` 文件，适合本地工具。
- **electron-vite**：构建工具，解决 Electron 多进程打包问题。

### 核心功能

1. **Markdown 导入**：解析半结构化 Markdown，提取题目、答案、追问、注意事项，自动跳过重复题。
2. **题库管理**：按来源文件管理题库，支持来源筛选、复习状态筛选、关键词搜索。
3. **复习模式**：普通复习（间隔重复）、错题复习（针对弱项）、突击模式（面试前快速过题）。
4. **复习统计**：首页展示已复习、未复习、错题、低掌握题、平均掌握度等指标。

### 难点

- **Markdown 半结构化解析**：面试手册格式不统一，需要设计容错的解析策略。
- **复习数据模型设计**：拆分 questions / review_progress / review_records 三表，支持多种复习模式。
- **动态筛选组合**：来源筛选 + 复习状态筛选 + 关键词搜索需要动态构建 WHERE 子句。
- **关联数据删除**：删除题库来源时，需要按正确顺序清理三个表的关联数据。

### 当前结果

- 已导入 155 道题，覆盖 AI 应用后端、Java 后端、项目经验三个方向。
- 支持 6 种复习状态筛选。
- 突击模式已用于面试前快速复习。
- 所有功能本地运行，无网络依赖。

## 3. 面试官可能追问

### Q1：为什么做桌面端而不是 Web？

**回答**：

主要是使用场景决定的。面试复习是个人行为，不需要多人协作，也不需要随时随地访问。桌面端的优势是：

- 本地运行，不依赖服务器，离线可用。
- 数据存储在本地，隐私性好，不需要注册账号。
- 启动快，不需要浏览器标签页。

如果做成 Web，需要部署服务器、处理用户认证、考虑数据同步，这些对于个人复习工具来说是过度设计。

### Q2：为什么用 Electron？

**回答**：

Electron 的优势是跨平台能力和 Web 技术栈复用。对于这个项目：

- 我熟悉 Vue + TypeScript，用 Electron 可以直接复用前端技术栈。
- Electron 提供了文件系统访问、原生对话框等桌面能力，这些是纯 Web 无法实现的。
- 虽然 Electron 打包体积较大，但对于桌面工具来说可以接受。

如果追求极致性能和小体积，Tauri 是更好的选择，但需要 Rust 生态，学习成本更高。

### Q3：为什么用 sql.js，而不是 SQLite native binding？

**回答**：

sql.js 是纯 JavaScript 实现的 SQLite，不需要编译原生模块。选择它的原因是：

- **Electron 打包简单**：原生模块（如 better-sqlite3）需要针对 Electron 版本预编译，配置复杂，容易出错。
- **部署简单**：纯 JS 依赖，npm install 即可，不需要 node-gyp。
- **数据可移植**：sql.js 可以将数据库导出为 Uint8Array，方便备份和迁移。

缺点是性能不如原生 binding，但对于 155 道题的本地工具来说，性能差异可以忽略。

### Q4：Markdown 是怎么解析成题目的？

**回答**：

我设计了一个半结构化解析器，核心逻辑是：

1. **按 `##` 分割**：每个二级标题标记一道新题。
2. **识别特殊标记**：`follow_ups:` 标记追问，`warnings:` 标记注意事项。
3. **答案收集**：题目和下一个特殊标记之间的内容作为答案。
4. **容错处理**：格式不完全符合规范时，尽量提取，而不是直接报错。

这种设计允许 Markdown 格式有一定的灵活性，不需要严格遵循模板。

### Q5：如何避免重复导入？

**回答**：

在导入时，对每道题生成一个唯一标识：`title + source_file`。

插入前先查询是否已存在，如果存在就跳过，并在导入报告中记录 `skippedCount`。

这样用户可以放心地多次导入同一个文件，不会产生重复数据。

### Q6：题库来源筛选是怎么设计的？

**回答**：

数据库中每道题都有 `source_file` 字段，记录它来自哪个 Markdown 文件。

来源筛选的实现是：

1. `getQuestionSources`：查询所有不重复的 `source_file` 及其题目数量。
2. `listQuestions` / `searchQuestions`：接受 `sourceFile` 参数，动态添加 `WHERE source_file = ?` 条件。

用户选择特定来源后，只显示该来源的题目。

### Q7：复习状态怎么计算？

**回答**：

复习状态基于 `review_progress` 表的字段计算：

- **未复习**：`review_count = 0`
- **已复习**：`review_count > 0`
- **错题**：`wrong_count > 0`
- **低掌握**：`review_count > 0 AND mastery_score < 60`
- **已掌握**：`review_count > 0 AND mastery_score >= 80`

`mastery_score` 是根据评分历史计算的掌握度分数（0-100）。当前采用最近表现的指数加权平均，高分会提升掌握度，低分也能将掌握度拉低；复习间隔按 1-5 分映射为 1、1、3、7、15 天。

### Q8：错题复习如何实现？

**回答**：

错题复习查询 `wrong_count > 0` 的题目，这些是用户在普通复习中评分为 1 或 2 的题。

进入错题复习后：

- 展示题目卡片，用户查看答案并自评。
- 提交评分走 `submitReview`，写入 `review_records` 并更新 `review_progress`。
- 如果评分 <= 2，`wrong_count` 会继续 +1。
- 如果评分 >= 3，`wrong_count` 不会自动减少。
- 如果用户认为已经掌握，需要通过 `resetWrongCount` 将 `wrong_count` 重置为 0。

这样把"继续复习"和"确认掌握"拆成两个动作，避免一次高分就自动移出错题池。

### Q9：突击模式为什么不写 review_records？

**回答**：

突击模式的定位是面试前快速过题，特点是：

- 时间紧迫，来不及仔细思考。
- 目的是快速回忆，不是深度学习。
- 评分可能不准确，不应影响正式的复习进度。

所以突击模式只展示题目，不记录评分，不更新 `review_progress`，不写入 `review_records`。这样突击时的数据不会污染正常的复习统计。

### Q10：删除题库来源时如何避免孤儿数据？

**回答**：

数据库启用了外键约束，业务层仍显式按正确顺序删除，并用事务保证三个步骤要么全部成功、要么全部回滚：

1. 先查出该来源所有题目的 `id` 列表。
2. 删除 `review_records` 中这些 `question_id` 的记录。
3. 删除 `review_progress` 中这些 `question_id` 的记录。
4. 最后删除 `questions` 表中该 `source_file` 的记录。

事务提交后只统一导出并持久化一次数据库文件，既保证一致性，也避免每删除一批记录就重复写整个文件。

### Q11：为什么没有接入 AI？

**回答**：

这是有意为之的设计决策：

- **核心需求明确**：题目导入、复习、统计这些功能不需要 AI。
- **避免过度设计**：接入 AI 需要处理 API Key 管理、网络请求、错误处理、成本控制，增加大量复杂度。
- **保持离线能力**：一旦依赖 AI，就失去了离线可用的优势。

但架构上预留了扩展点：后续可以为每道题增加 AI 分析字段，作为可选功能。

### Q12：如果题库从 155 道变成 5000 道，你会怎么优化？

**回答**：

当前设计有 500 条加载上限，是为了防止大量数据一次性加载导致页面卡顿。如果题库增长到 5000 道，我会从几个方向优化：

1. **虚拟列表**：只渲染可视区域的题目，大幅减少 DOM 节点。
2. **服务端分页**：将分页逻辑移到数据库层，每次只查询当前页的数据。
3. **索引优化**：为 `source_file`、`review_count`、`mastery_score` 等常用筛选字段添加索引。
4. **懒加载**：题目详情（答案、追问）在展开时才加载，而不是列表加载时一起查。

当前架构已经做了分层，这些优化可以逐步进行，不需要重构。

## 4. 项目亮点

### 技术亮点

- **本地优先架构**：不依赖后端服务，数据完全在本地，启动即可用，离线无影响。
- **Markdown 半结构化解析**：设计了容错的解析策略，允许 Markdown 格式有一定灵活性。
- **复习数据模型设计**：拆分 questions / review_progress / review_records 三表，支持多种复习模式互不干扰。
- **动态筛选组合**：来源筛选 + 复习状态筛选 + 关键词搜索可以组合使用，通过动态 WHERE 子句实现。
- **关联数据安全删除**：删除题库来源时，按正确顺序清理三个表的关联数据，避免孤儿记录。

### 工程亮点

- **Electron 安全规范**：使用 contextBridge 暴露 API，Renderer 不直接访问 ipcRenderer。
- **统一错误处理**：所有 IPC 返回 `{ success, data?, error? }` 格式，Renderer 统一处理。
- **纯文本展示**：不使用 v-html 渲染答案，避免 XSS 风险。
- **危险操作二次确认**：删除题库来源等不可逆操作，有二次确认弹窗。
- **导入报告**：导入完成后展示详细报告，让用户知道导入了多少、跳过了多少。

### 产品亮点

- **功能闭环**：导入 → 搜索 → 复习 → 统计 → 管理，形成完整闭环。
- **多种复习模式**：普通复习、错题复习、突击模式，覆盖不同场景。
- **复习状态可视化**：首页统计让用户清楚知道复习进度。

## 5. 可以优化的地方

保持诚实，面试时可以说这些是已知的改进方向：

- **缺少单题编辑**：当前只能通过重新导入覆盖，无法单独修改某道题。
- **Markdown 兼容性有限**：只支持特定格式的面试手册，复杂 Markdown 可能解析不完整。
- **缺少虚拟列表**：题目数量多时，列表渲染可能卡顿。
- **缺少数据导出**：无法将复习数据导出为 CSV 或其他格式。
- **复习算法简单**：当前掌握度采用指数加权，间隔使用固定映射，不如专业间隔重复算法（如 Anki 的 SM-2）精确。
- **自动化测试仍需扩充**：已覆盖 Markdown 核心解析和掌握度计算，后续还需要覆盖 IPC 与数据库事务回滚。
- **缺少分页控件**：虽然有分页逻辑，但 UI 上没有分页导航。

这些不是缺陷，而是"已知的优化方向"，说明你对项目有清晰的认识。

## 6. 讲项目时的注意事项

### 不要说

- ❌ "这是一个 AI 面试助手"
- ❌ "已经接入了大语言模型"
- ❌ "支持云同步和多端协作"
- ❌ "可以自动发布安装包"

### 要说

- ✅ "这是一个本地面试复习工具"
- ✅ "解决 Markdown 面试资料的管理和复习问题"
- ✅ "基于 Electron + Vue 3 + TypeScript + sql.js"
- ✅ "所有数据存储在本地，离线可用"
- ✅ "架构上预留了 AI 扩展点，但当前版本不依赖 AI"

### 讲解节奏建议

1. **30 秒**：一句话说清楚项目是什么、解决什么问题。
2. **1 分钟**：展开讲核心功能和技术选型。
3. **2 分钟**：讲难点、设计决策、项目亮点。
4. **按需展开**：根据面试官追问，深入讲解具体技术细节。

不要一上来就讲技术细节，先让面试官理解项目的价值和定位。
